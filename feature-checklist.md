## 기능 체크리스트
구현과 테스트를 종합적으로 고려하여<br>
미션의 요구사항을 수행했다 판단되면 ✅로 표시합니다.

### 차량
- [x] 자동차는 이름을 갖는다.
- [x] 자동차는 움직일 수 있다,
    - [x] 0 ~ 9 사이의 random 값을 구한 후, 4 이상일 경우 전진하고 3 이하일 경우 멈춘다.

### 게임
- [x] n대의 자동차가 참여할 수 있다.
- [x] 주어진 횟수 동안 자동차는 전진 또는 멈출 수 있다.
  - [x] 0 ~ 9 사이의 random 값을 구한 후, 4 이상일 경우 전진하고, 3 이하의 값이면 멈춘다.
- [x] 자동차 경주 게임을 완료한 후 누가 우승했는지를 구할 수 있다. 우승자는 한 명 이상일 수 있다.
- [x] 각 자동차에 이름을 부여할 수 있다. 전진하는 자동차를 출력할 때 자동차 이름을 같이 출력한다
- [x] 자동차 이름은 쉼표를 기준으로 구분하며, 이름은 5자 이하만 가능하다.
- [x] 사용자는 몇 번의 이동을 할 것인지 입력할 수 있어야 한다.
- [x] 모든 로직에 단위 테스트를 구현한다.
  - 단, `System.in`, `System.out` 로직은 제외한다.
  - 랜덤한 요소가 존재하는 코드는 어떻게 테스트할 수 있는지 경험한다.

## 26.08.22 리뷰 반영 체크리스트
### `MovableCar` 네이밍
Car라는 포괄적인 개념 자체는 `움직인다`, `정지해있다` 두가지 행동을 할 수 있다고 당연하게 여겨지고 있습니다.<br>
이때문에 대길님께서 `Movable`이라는 것이 객체를 잘 설명하지 못하였다고 이야기해주신 것 같다는 생각이 들었습니다.

대길님께서 처음에도 강조하셨던 단순히 "정답인 것처럼 이야기하지 않겠다"라는 입장 저도 정말 몇번이고 생각하는 부분이라 잘 기억하고 있습니다만,
`RacingCar`라는 이름이 이 게임에서 사용되는 차량을 직관적으로 잘 설명할 수 있는 것 같아서 이 네이밍으로 수정하게 되었습니다
<br><br>
- [x] `MovableCar` -> `RacingCar`로 수정 완료

### 입력을 담당하는 `InputView`와 초기화를 담당하는 `ResultView`
제 기저에는 FE는 보통 Javascipt를 사용하고, Javascript는 싱글스레드이므로 데이터 관련 검증은 전부 BE에서 한다.<br>
라고 막연히 생각하고 있어서, Java에서 입력 -> 비즈니스 로직에서 사용할 형태로 매핑 -> 비즈니스 로직에 전달의 형태에서<br>
입력 단계에서 유효한 형태로 모두 가다듬고 차량 초기화는 완벽히 유효한 입력값들로만 사용했었습니다.

하지만, 사용자와 상호작용하는 View와 비즈니스 로직을 수행하는 domain,<br>
그리고 domain에서 사용하기 알맞은 형태로 매핑하는 단계가 있음을 리뷰를 통해 알게되었고, `CarListInitializer`에서 검증을 하도록 수정했습니다.
<br><br>
- [x] `InputView`에서 중복, 5글자 제한 검증 -> `CarListInitializer`에서 중복, 5글자 제한 검증
- [x] `CarListInitializer`에서 차량 이름이 빈 문자열로 입력되었을 시, 재입력 요청하는 로직으로 수정

### `getThreshold()` 미사용
Move 인터페이스에서 Threshold를 변수로 입력받고, 임계값을 처리하는 로직에 threshold 이상이면 전진, 미만이면 정지 로직을 리팩토링하려고 했었으나,<br>
깜빡하고 리팩토링하지 않아 미사용으로 남아있었습니다. 이 점 수정했습니다
<br><br>
- [x] getThreshold로 전진, 정지 테스트를 더욱 유연하게 처리
- [x] min, max 필드를 Move 인터페이스에 작성하여, RacingGameApplication의 최상단에서 상수 관리, 테스트 역시 상수로 관리
